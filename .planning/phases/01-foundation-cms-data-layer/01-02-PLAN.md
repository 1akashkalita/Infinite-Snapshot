| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | Stale/rotated dataset IDs silently capturing the wrong data | mitigate | Re-resolve all three dataset IDs against the CMS metastore at capture time and throw if any is absent (D-03 / rule #3). |
| T-02-02 | Information disclosure | Outbound request to a user-controlled host (SSRF) | accept | No user-controlled URLs — the capture script only fetches a fixed allowlist of three CMS dataset IDs plus the metastore. Dev-time only; not exposed as a route. |
| T-02-03 | Denial of service | CMS API returns an error or empty result, producing a broken/empty fixture | mitigate | queryCMS throws on non-ok HTTP status and on count === 0, so a failed capture fails loudly instead of writing an empty fixture. |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | No installs in this plan; the package install + legitimacy checkpoint is handled in Plan 01-01 (T-01-SC). No new dependencies are added here (native fetch + already-installed tsx). |

No HIGH-severity threats. No secrets, auth, or PII; the only network egress is to the fixed public CMS allowlist, dev-time only.

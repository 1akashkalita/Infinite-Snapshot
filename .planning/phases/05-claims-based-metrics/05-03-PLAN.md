<objective>
Make `hospMetrics` a real, Zod-validated field inside the canonical `ReportViewModelSchema` (D-13), thread it through `assembleViewModel`, and extend `GET /api/facility` to fan out to all three datasets via `Promise.allSettled` (D-07/D-08), returning provider data + `hospMetrics` (or, on a claims/averages fetch **rejection**, provider data alone) in one response.

Purpose: This is the server-side plumbing that delivers CLM-01 end-to-end at the data boundary. The view-model change is load-bearing for the PDF: `POST /api/export/pdf` re-validates the full posted model against `ReportViewModelSchema`, so `hospMetrics` MUST live inside it (D-13 / RESEARCH Pitfall 3). The route change implements the graceful-degradation contract: provider info is the only hard dependency (D-07); a claims/averages fetch **rejection** degrades to an absent `hospMetrics` (D-09), while a fetch that *succeeds* with partial data (fewer than 4 measures) still yields the full 12-row array with per-row suppression (D-10/SC#5 — that logic lives in the mapper, Plan 02).

Output: 1 modified view-model + 1 modified route, plus 2 extended tests (view-model, facility API).
</objective>
